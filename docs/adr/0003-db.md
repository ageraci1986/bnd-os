# ADR 0003 — Base de données : Supabase (PostgreSQL 16)

- **Statut :** Accepté
- **Date :** 2026-04-27
- **Décideur :** Angelo L.

## Contexte

NexusHub a besoin de :

- PostgreSQL (multi-tenant via RLS)
- Realtime broadcast (Kanban, checklist, communications)
- Storage (avatars, pièces jointes V1.5)
- Edge functions optionnelles
- Backups automatiques + PITR

## Options évaluées

| Critère                       |  Supabase   |   Neon    | RDS / self-hosted |
| ----------------------------- | :---------: | :-------: | :---------------: |
| Postgres managé               |     ✅      |    ✅     |        ✅         |
| Auth intégré                  |     ✅      |    ❌     |        ❌         |
| Realtime intégré              |     ✅      |    ❌     |        ❌         |
| Storage intégré               |     ✅      |    ❌     |        ❌         |
| Branching (DB ephemeral)      |   ⚠️ Bêta   |    ✅     |        ❌         |
| Coût V1 (5–20 users / agence) |  ~25$/mois  | ~20$/mois |     50$+/mois     |
| RLS Postgres natif            |     ✅      |    ✅     |        ✅         |
| Backups + PITR                | ✅ Pro plan |    ✅     |      manuel       |

## Décision

**Supabase** est retenu : il fournit **DB + Auth + Realtime + Storage** dans un seul service, avec un coût prévisible et une intégration native. La consolidation simplifie l'opérationnel pour une équipe restreinte (V1).

## Setup imposé

- Plan : **Pro** (PITR + backups quotidiens + bandwidth)
- Région : **eu-central-1** (Francfort) ou **eu-west-3** (Paris) — RGPD
- 2 projets Supabase :
  - `nexushub-staging`
  - `nexushub-prod`
- ORM : **Prisma 6** sur le DATABASE_URL Supabase
  - Migrations versionnées (`prisma/migrations/`)
  - Connection pooling : URL "transaction" Supabase (PgBouncer)
- Schémas Postgres :
  - `public` : tables applicatives
  - `auth` : géré par Supabase Auth (ne pas modifier)
  - `audit` : logs append-only

## RLS (Row Level Security)

**Activée sur 100% des tables applicatives** dès la création. Politiques par défaut :

```sql
-- Policy générique multi-tenant
CREATE POLICY "Workspace members can read"
  ON public.<table> FOR SELECT
  USING (workspace_id IN (
    SELECT workspace_id FROM public.memberships WHERE user_id = auth.uid()
  ));

CREATE POLICY "Workspace members can write"
  ON public.<table> FOR INSERT WITH CHECK (
    workspace_id IN (SELECT workspace_id FROM public.memberships WHERE user_id = auth.uid())
  );
-- + UPDATE / DELETE selon le rôle (Admin / Member)
```

Tests automatiques en CI : `psql` se connecte en tant que rôle `anon` puis `authenticated` (différents JWT) et vérifie qu'aucune fuite cross-workspace n'est possible.

## Backups & DR

- Backups quotidiens automatiques (7 jours rétention plan Pro)
- PITR (Point-In-Time Recovery) sur 7 jours
- Procédure de restore documentée dans `docs/runbooks/db-restore.md`

## Conséquences

- Variable env : `DATABASE_URL` (URL session) + `DIRECT_URL` (URL direct, pour migrations Prisma)
- Variables Supabase : `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`
- Lint custom : interdire `prisma.<model>.findMany()` sans clause `where: { workspace_id }` (Phase 2)
