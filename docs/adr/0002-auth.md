# ADR 0002 — Solution d'authentification : Supabase Auth

- **Statut :** Accepté
- **Date :** 2026-04-27
- **Décideur :** Angelo L.

## Contexte

NexusHub V1 doit gérer :

- Login email + mot de passe
- Reset password
- Invitations par email avec lien signé (flow custom : token, expiration 72h, single-use)
- Sessions sécurisées
- RLS Postgres multi-tenant
- 2FA prévu en V1.5

Trois options évaluées :

| Critère                             |        Supabase Auth         |   Auth.js v5   |   Better-Auth    |
| ----------------------------------- | :--------------------------: | :------------: | :--------------: |
| Intégration native PostgreSQL + RLS |              ✅              |  ⚠️ (manuel)   |   ⚠️ (manuel)    |
| Maturité 2026                       |          ✅ Mature           |   ✅ Mature    | ⚠️ Jeune (~1 an) |
| Coût                                | ✅ Inclus dans plan Supabase | ✅ Open source |  ✅ Open source  |
| Magic-link / OTP                    |              ✅              |       ✅       |        ✅        |
| MFA TOTP                            |           ✅ Natif           |   ⚠️ Adapter   |        ✅        |
| JWT vérifiable côté serveur         |              ✅              |       ✅       |        ✅        |
| Couplage à Postgres pour RLS        |   ✅ Direct (`auth.uid()`)   |  ❌ Faux JWT   |   ❌ Faux JWT    |
| Risque vendor lock-in               |           ⚠️ Moyen           |   ✅ Faible    |    ✅ Faible     |

## Décision

**Supabase Auth** est retenu pour V1 :

1. Couplage direct avec Supabase Postgres → RLS via `auth.uid()` natif, sans table de jointure manuelle.
2. Mature, documenté, audité.
3. SMTP custom (Resend) supporté pour brander les emails.
4. MFA TOTP prêt pour V1.5 sans refonte.
5. Le flow d'invitation custom (token HMAC + expiration 72h + audit) reste **côté NexusHub** : Supabase est utilisé uniquement pour `admin.createUser()` et `signInWithPassword()` côté serveur après validation du token.

Le risque de lock-in est jugé acceptable car le schéma Postgres reste standard (table `auth.users` standard, exportable). En cas de migration future, un script peut copier `auth.users` vers une autre solution.

## Configuration imposée

- `JWT expiry = 3600s` (1h)
- `Refresh token rotation = ON`, `reuse interval = 10s`
- `Enable signup = OFF` (inscription publique désactivée)
- `Email confirmations = OFF` (notre flow d'invitation valide déjà l'email)
- SMTP custom Resend (templates FR/EN)
- Service role key **uniquement côté serveur** (jamais exposée client)
- Anon key publique (autorisée côté client) → toutes les opérations critiques passent par RLS

## Conséquences

- `@supabase/ssr` côté Next.js pour gérer cookies et session.
- `requireUser(req, role?)` middleware utilise `supabase.auth.getUser()` (vérification JWT) avant de charger le `Membership`.
- Les Server Actions critiques exigent CSRF token en plus.
- Les variables `NEXT_PUBLIC_SUPABASE_URL` et `NEXT_PUBLIC_SUPABASE_ANON_KEY` sont publiques (RLS protège).
- `SUPABASE_SERVICE_ROLE_KEY` est **secret**, utilisé uniquement dans des contextes server-only (`'server-only'` import).

## Risques & mitigations

- **Risque** : fuite de la `service_role` key → mitigation : import `server-only`, scan gitleaks, lint custom interdisant l'import dans `'use client'`.
- **Risque** : politique RLS mal écrite → mitigation : tests automatiques par rôle (anon, authenticated, service_role) en CI.
