# ADR 0005 — Jobs background : Inngest

- **Statut :** Accepté
- **Date :** 2026-04-27

## Contexte

NexusHub a besoin de jobs background fiables :

- **Toutes les heures** : scan échéances dépassées → `moveToBlocked` (cf. CLAUDE.md §6.3)
- **Quotidien** : archivage cartes 30j en dernière colonne (opt-in projet)
- **À la demande** : envoi mail invitation via Resend (latence > 200ms)
- **Sur événement** : refresh OAuth tokens Slack / Graph avant expiration
- **Webhooks** : ingestion messages Slack / Graph (haute volumétrie possible)

Contraintes : retry, idempotence, observabilité, secrets safe.

| Crons                    |      ✅      |      ✅      |       ✅       |
| TypeScript first-class   |      ✅      |      ✅      |       ✅       |

## Décision

**Inngest** retenu :

- Modèle "step functions" idéal pour l'auto-archivage et le scan d'échéances (steps idempotents par carte)
- UI dev locale pour debug (`inngest dev`)
- Retry + dead-letter natifs
- Free tier (50K runs/mois) couvre largement les besoins V1

## Jobs définis (Phase 5–6)

| Job                          | Trigger                             | Description                                             |
| ---------------------------- | ----------------------------------- | ------------------------------------------------------- |
| `cards.scan_overdue`         | Cron `0 * * * *` (chaque heure)     | Liste les cartes overdue, fan-out un step par workspace |
| `cards.archive_done`         | Cron `0 3 * * *` (3h du matin)      | Archive les cartes Done > 30j si projet opt-in          |
| `email.send_invitation`      | Event `invitation.created`          | Envoi via Resend, retry 3 fois                          |
| `oauth.refresh_token`        | Cron `0 */6 * * *`                  | Refresh Slack + Graph tokens proches expiration         |
| `slack.process_event`        | Event `slack.event_received`        | Traite un événement Slack (idempotence sur `event_id`)  |
| `graph.process_notification` | Event `graph.notification_received` | Idem Graph                                              |

## Sécurité

- Signature des webhooks Inngest vérifiée (`INNGEST_SIGNING_KEY`)
- Steps **idempotents** : retry sans effet de bord
- Logs sans PII (cf. CLAUDE.md §4.7)

## Conséquences

- Variable env : `INNGEST_EVENT_KEY`, `INNGEST_SIGNING_KEY`
- Endpoint Next.js `/api/inngest` exposé pour les workers
- Tests : Inngest fournit un harness pour tester les step functions
